# Скилл: Clone Step 3 — Написание ТЗ (спека)

## Триггер

Когда анализ сервиса завершён и нужно превратить его в структурированное задание для реализации.

## Что делает

Создаёт PROJECTCLONE/README.md — документ, по которому можно собрать MVP без дополнительных уточнений.

## Структура спека

```
## Выбранный сервис
Название, URL, почему выбран, гипотеза о moat.

## Необходимые скиллы
Список инструментов (Claude Code / Cursor / Supabase / etc.)

## Продуктовые модули
Таблица: модуль — что делает — каким скиллом реализуется

## Логика сборки
- Как создавали (порядок шагов)
- Что было самым сложным
- Открытые вопросы

## Ров
По 2–3 типам: данные / switching cost / регуляторка / сеть — с конкретной стратегией.
```

## Принципы написания

- Модули: только то, что входит в MVP. Срезать всё, что не нужно для первого запуска.
- Стек: зафиксировать явно, не оставлять "разберёмся по ходу"
- Ров: не абстрактно ("данные важны"), а конкретно ("хранить audit_log с day 1, версионировать протоколы")

## Контекст для подкладки

- Результат анализа из `clone-2-analysis` (стек, список экранов, moat)
- Временной горизонт MVP

## Пример

Benchling → спек Квантары: 7 модулей (Dashboard, Projects, Gantt, Reagents, Consumables, Profile, Auth), стек React+Supabase, три рва (данные / switching cost / GxP).

**Следующий шаг:** передать спек в скилл `clone-4-implementation`.
