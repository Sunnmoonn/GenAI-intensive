# Скилл: Project Clone Workflow

## Триггер

Когда нужно найти зарубежный сервис для клонирования и довести его до рабочего MVP.

## Пошаговая методология

### Шаг 1 — Поиск кандидата

1. Запустить `startup-market` — скрейпер YC/ProductHunt/Crunchbase собирает актуальные компании
2. Отфильтровать по нише: B2B SaaS, отраслевой вертикали, барьерам для западных игроков в РФ
3. Применить VENTURE-SCAN RF 2.0 скоринг (формула `(M² × T × A × B × P)^(1/6) × 10`) — автоматически считается в `lib/scoring.ts`
4. Топ-10 по скору → ручной просмотр: есть ли боль, есть ли рынок в РФ, есть ли аналоги
5. Финальный выбор: максимальный скор + отсутствие сильного российского конкурента

**Критерии отсева:** западный сервис заблокирован / не работает в РФ, рынок > ₽500 млн, нет прямого российского аналога с полным функционалом.

### Шаг 2 — Анализ интерфейса и архитектуры

1. Скрейпить сайт через Firecrawl: главная, pricing, docs, changelog → конкурентная матрица
2. Прогнать через скилл `benchling-arch-research` — извлекает frontend/backend/data/infra по публичным источникам (engineering blog, job postings, API docs)
3. Составить список экранов для MVP: что реально нужно, что можно срезать на первой итерации
4. Зафиксировать moat: где реальное конкурентное преимущество — в данных, switching cost или регуляторке

### Шаг 3 — Написание спека

Структура `PROJECTCLONE/README.md`:
- Выбранный сервис + обоснование (почему он, не другой)
- Таблица модулей: что делает каждый экран
- Таблица скилов: какими инструментами реализуется
- Архитектура: стек (frontend / backend / auth / БД)
- Ров: стратегия удержания на 3–6 месяцев

### Шаг 4 — Имплементация

| Задача | Инструмент |
|---|---|
| React/TypeScript компоненты по описанию экрана | Claude Code |
| Рефакторинг, типы, адаптация под проект | Cursor |
| Схема БД, RLS-политики, Auth, Edge Functions | Supabase (через Claude Code) |
| Мобильная адаптация | Claude Code (итеративно) |

Порядок: сначала схема БД + Auth → потом страницы в порядке приоритета → мобильная адаптация последней.

### Шаг 5 — Тестирование

1. Vercel Preview — каждый PR автоматически получает preview URL
2. Сценарии вручную: регистрация → создание организации → добавление данных → проверка RLS (данные не утекают между org)
3. Мобильный вид: Chrome DevTools + реальное устройство
4. Проверка edge cases: пустые состояния, ошибки сети, просроченная сессия

### Шаг 6 — Публикация

1. `vercel.json` с SPA rewrite rules (`"routes": [{"src": "/(.*)", "dest": "/index.html"}]`)
2. Суpabase: включить RLS на всех таблицах, задеплоить Edge Function
3. GitHub: убрать `.env.local`, `.vercel/`, `node_modules/` через `.gitignore`
4. Vercel: подключить GitHub repo → Production deployment → проверить env vars

## Контекст для подкладки

- Название сервиса + URL
- Целевой рынок (B2B / B2C, ниша)
- Бюджет времени на MVP (влияет на список модулей)
- Стек (по умолчанию: React + TypeScript + Vite + Tailwind + Supabase)

## Пример применения

Benchling (ELN+LIMS) → Квантара: найден через анализ рынка импортозамещения после санкций 2024 г., проанализирован через `benchling-arch-research`, реализован за ~2 недели, задеплоен на Vercel. Moat: накопленные данные протоколов + switching cost + GxP-сертификация.
