# PODGOTOVKA_K_RAZRABOTKE

## 1. Цель этапа подготовки

Перед тем как генерировать код, нужно зафиксировать: что именно строим, зачем, в каком объёме и какими инструментами. Без этого этапа велик риск собрать либо слишком большой клон (не успеть), либо слишком маленький (потерять продуктовую глубину). Подготовка закрывает три вопроса: **что клонируем** (JTBD + сценарий), **как разложено** (модули + scope) и **чем строим** (skills + стек).

---

## 2. Выделение core JTBD

### Выбранная core JTBD

> «Когда мой близкий умирает, я хочу уведомить все организации об этом факте — не обзванивая каждую отдельно — и отслеживать статус каждого запроса, чтобы закрыть все административные обязательства и ничего не упустить.»

Это объединённая JTBD из двух неразрывных частей: **уведомить** (действие) и **отследить** (контроль результата). Вместе они образуют замкнутый цикл с чётким началом и концом.

### Почему выбрана именно она

- **Замкнутость:** есть чёткое начало (смерть → форма), процесс (отправка запросов) и конец (все организации ответили). Большинство других JTBD в продукте не имеют такого законченного цикла.
- **Репрезентативность:** именно этот сценарий отличает Settld от простого todo-приложения. State machine с переходами статусов — это core value, а не декорация.
- **Реализуемость:** не требует реальных API-интеграций с организациями. Логика воспроизводима через mock-данные и ручную смену статусов.

### Альтернативные JTBD, сознательно не взятые в MVP

| JTBD | Почему не взята |
|------|----------------|
| «Понять, что вообще нужно сделать» (чеклист/guidance) | Контентный слой, не продуктовый. Заменяется статичным текстом |
| «Хранить и передать важные документы» | Требует file storage, прав на файлы — отдельный модуль без влияния на core-сценарий |
| «Обработать входящий запрос как организация» | Входит в MVP как Org Portal, но не является самостоятельной JTBD для инициатора |
| «Снять эмоциональную нагрузку» | Это tone-of-voice и UX-решения, а не отдельный функциональный модуль |

## 3. Выбор основного пользовательского сценария

### Название сценария

**«От смерти до дашборда»** — полный цикл от создания кейса до отслеживания статусов.

### Пошаговый flow

```
[Initiator]
1. Регистрация / вход
2. Wizard: данные умершего (шаг 1)
3. Wizard: данные инициатора + роль (шаг 2)
4. Wizard: подтверждение, создание DeathCase (шаг 3)
5. Выбор организаций по категориям → создание NotificationRequest per Org
6. Case Dashboard: прогресс-бар + список задач со статусами

[Organisation]
7. Вход в Org Portal → Inbox входящих запросов
8. Открытие запроса → смена статуса (Acknowledged / Processing / Completed / Rejected)

[Initiator]
9. Dashboard обновляется → инициатор видит прогресс в реальном времени
10. Когда все запросы в Completed → кейс закрыт
```

### Почему именно этот сценарий

- Участвуют **две роли** с разными интерфейсами — показывает ролевую модель и разграничение доступа.
- **State machine работает интерактивно** через UI, а не через скрипт — демонстрирует бизнес-логику, а не только CRUD.
- Сценарий **самодостаточен**: его можно показать от начала до конца без объяснений «а здесь будет...».

## 5. Выбор scope для учебного клона

### Выбранный вариант: Оптимальный

**7 модулей, 9–10 экранов, 2 роли.**

```
Initiator side:     Login → Wizard(×3) → Org Picker → Dashboard → Task Detail
Organisation side:  Login → Org Inbox → Request Detail (кнопки статусов)
```

### Что вошло в MVP

Auth & Roles, Death Case Wizard, Organisation Registry, Notification Engine, State Machine, Case Dashboard, Org Portal.

### Что сознательно за рамками

Documents, Email/Push, Guidance, Admin Panel, Real Org Integrations, Analytics, Billing.

### Почему такой scope оптимален

**Минимальный вариант** (без Org Portal) технически проще, но state machine не работает интерактивно — преподаватель видит незаконченную систему.

**Расширенный вариант** (+ Documents + Email) добавляет ~40% объёма работы без продуктовой глубины — это инфраструктурный overhead, повышающий риск не успеть.

**Оптимальный вариант** — минимальный набор, при котором продукт работает как продукт: полный цикл, две роли, живая state machine. Убери любой из 7 модулей — и сценарий рвётся.

## 6. Skills, которые использовались на этапе подготовки

### `jtbd-extractor`

**Зачем:** структурировать работу пользователя в формате JTBD, выбрать одну главную для MVP и развернуть её в пошаговый user flow с ролями.

**Результат:** сформулирована core JTBD («уведомить всё и отследить статусы»), выбрана как основа MVP, определены роли (Initiator / Organisation), построен 10-шаговый flow, зафиксированы сознательные исключения из scope.

---

### `product-architecture-mapper`

**Зачем:** разложить продукт на независимые модули, классифицировать как CORE/SUPPORTING/OPS, построить три уровня scope и матрицу «модуль → навыки для реализации».

**Результат:** выявлено 14 модулей полного продукта, сформированы три варианта scope (Minimal / Optimal / Extended), выбран Optimal из 7 модулей, построена матрица module → skill для всех модулей MVP.

---

### `cloneability-assessment`

**Зачем:** системная оценка Settld как кандидата для клонирования по 10 критериям до начала проектирования.

**Результат:** Settld получил 49/50 — максимальный балл среди 20 кандидатов. Подтверждена обоснованность выбора: чёткий сценарий, реалистичный MVP, богатый moat, нестандартная тема.

---

### `moat-designer`

**Зачем:** структурировать анализ конкурентного рва для стратегической части домашнего задания.

**Результат:** определены ключевые источники рва Settld: сетевые эффекты (org-side), регуляторные партнёрства, brand trust, data moat. Зафиксированы уязвимости и что углубляет ров при масштабировании.

## 7. Итог подготовки к разработке

### Что стало понятно

- **Что строим:** MVP-клон вокруг одного замкнутого сценария с двумя ролями и живой state machine.
- **Что не строим и почему:** 7 модулей за рамками — не потому что сложно, а потому что они не добавляют продуктовой глубины в учебном контексте.
- **Какова архитектура:** data model с 5 сущностями (User, DeathCase, Organization, NotificationRequest, StatusEvent), state machine с 6 переходами, role-based routing.
- **Чем строим:** Next.js + Tailwind + Prisma + SQLite — стек без внешних зависимостей, разворачивается одной командой.

### Почему можно переходить к генерации MVP

Все входные данные для `frontend-mvp-builder` готовы: сервис определён, JTBD зафиксирована, user flow построен, модули выбраны, scope зафиксирован, data model понятна. Нет открытых продуктовых вопросов, которые потребовали бы возврата к исследованию.

### Открытые вопросы и ограничения

| Вопрос | Статус |
|--------|--------|
| Реальные API-интеграции с организациями | Сознательно за рамками MVP; заменены ручной сменой статуса |
| Email-уведомления при смене статуса | За рамками MVP; пользователь проверяет статусы через dashboard |
| Фильтрация org в реестре (поиск) | В v1 — статичный список с категориями; поиск — v2 |
| Моат на российском рынке | Открытый вопрос для стратегической части задания |
