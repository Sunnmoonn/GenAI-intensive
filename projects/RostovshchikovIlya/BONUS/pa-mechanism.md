# PA_Mechanism — технология и бизнес-модель

## Когда запускать

Второй шаг после `PA_Surface`. Нужна заполненная surface-карточка — из неё берём core loop и сегменты.

## Что делает

Разбирает внутренности продукта: как он технически устроен и как зарабатывает. Отвечает на вопрос «почему вообще возможно то, что видит пользователь».

## Шаги

1. **Технологическая механика.** Что должно работать внутри, чтобы core loop из surface-карточки случился? Собрать 3–6 ключевых компонентов (API, ML-модель, hardware, база данных, интеграция). Для каждого — роль в loop.
2. **Точки отказа.** Где система хрупка? Что случится, если убрать один компонент?
3. **Источники данных / контента.** Откуда берётся топливо для продукта: пользовательский UGC, парсинг, ручная разметка, партнёрства, железо. Для каждого — стоимость привлечения единицы.
4. **Бизнес-модель.** Способ монетизации (подписка, % от транзакции, one-time, B2B-контракт, реклама). Средний чек. Цикл оплаты.
5. **Unit-экономика (гипотеза).** CAC, LTV, маржа на единице. Если нет данных — оценка и диапазон.
6. **Каналы продаж.** Где продукт находит пользователя. B2C vs B2B распределение.
7. **Регуляторный / compliance-контекст.** Что может сломать продукт со стороны государства/платформ.

## Выход

```
# <Продукт> — Mechanism

## Технологическая механика
| Компонент | Роль в loop | Хрупкость |
| ... | ... | ... |

## Источники топлива
- ...

## Бизнес-модель
- Тип: ...
- Цена: ...
- Цикл: ...

## Unit-экономика (гипотеза)
- CAC ≈ ...
- LTV ≈ ...
- Маржа ≈ ...

## Каналы
- ...

## Регуляторные риски
- ...
```

## Правила

- Если unit-экономика неизвестна — оценка с явным `(гипотеза)`, не скрывать неопределённость.
- Технологические компоненты описывать так, чтобы не-технарь понял назначение, а технарь — слой абстракции.
- Каналы описывать как измеримые источники (соц. сети, отраслевые события, SEO), а не «сарафанное радио в целом».
