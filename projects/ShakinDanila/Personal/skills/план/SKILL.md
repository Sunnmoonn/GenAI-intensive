# Скилл: Календарный план с часами и бюджетом

## Что генерирует скилл

1. **Команда и роли** — участники, их роли, нагрузка, зоны ответственности
2. **Риски и открытые вопросы** — реестр рисков с вероятностью и митигацией
3. **KPI и критерии приёмки** — измеримые показатели успеха с методом проверки
4. **Состав работ** — список рабочих пакетов
5. **Календарный план по неделям** — таблица этапов с часами по ролям и бюджетом
6. **Описание этапов** — детализация каждого этапа + конкретный результат (Deliverable)
7. **Итоги** — срок, трудоёмкость, бюджет, помесячная разбивка
8. **Контрольные точки** — ключевые milestone'ы с датами

---

## Шаг 1 — Сбор данных (AskUserQuestion)

Если информации недостаточно, задай всё сразу одним вопросом:

1. **Название и суть проекта** — что разрабатываем (1-2 предложения)
2. **Команда** — кто участвует, сколько человек, их роли
3. **Ставка / бюджет** — сколько стоит час или месяц работы каждого (или общий бюджет)
4. **Длительность** — есть ли дедлайн, или нужно оценить самостоятельно
5. **Нагрузка** — сколько часов в неделю у каждого участника
6. **Этапы** — если уже известны, перечислить; если нет — определяем по типу проекта
7. **Есть ли ТЗ или список задач** — прикрепить файл или описать кратко
8. **Формат вывода** — markdown-файл (по умолчанию) или текст в чате

Если информация уже есть в контексте разговора — не переспрашивай, сразу к шагу 2.

---

## Шаг 2 — Определить структуру этапов

По типу проекта предложи стандартный набор этапов, покажи план и спроси: «Генерировать? Или скорректировать?»

### Типовые наборы этапов

**Разработка ПО/AI:**
Инициация → Анализ/Архитектура → Инфраструктура → Backend → ML/CV-модуль → Frontend/UI → Интеграции → Тестирование → Пилот → Запуск

**Веб-платформа/SaaS:**
Инициация → Дизайн/UX → Архитектура → Backend+API → Frontend → Интеграции → QA → Запуск

**Аппаратный проект:**
Инициация → Проектирование → Закупка/инфраструктура → Сборка → Программная часть → Интеграция → Испытания → Запуск

**Внутренний инструмент/автоматизация:**
Анализ процессов → Прототип → MVP → Тестирование → Внедрение

**Произвольный проект:** определи этапы самостоятельно по описанию задач.

## Шаг 3 — Оценить часы и бюджет

### Принципы распределения часов

- Суммарная нагрузка каждого участника = одинакова на всём проекте (например, 40 ч/нед)
- Роли внутри недели меняются по этапам (PM больше на старте и финише, Dev — в середине)
- QA нарастает к концу, пик на этапе тестирования

### Распределение по ролям внутри недели (типовое)

| Роль | Инициация | Разработка | Тестирование | Запуск |
|------|-----------|-----------|--------------|--------|
| PM | 50-60% | 25-30% | 20% | 40-50% |
| Dev/Backend | 30-40% | 70-75% | 40-50% | 30-40% |
| CV/ML | 20-30% | 70-80% | 30-40% | 20-30% |
| QA | 10-20% | 15-25% | 50-60% | 30-40% |

### Расчёт бюджета этапа

```
Бюджет_этапа = (ставка₁ × часы_участника₁ за этап) + (ставка₂ × часы₂ за этап) + ...
```

Если ставка = месячная зарплата:
```
часовая_ставка = зарплата / (недели_в_месяце × часов_в_неделю)
```

Например: 50 000 руб/мес ÷ (4.33 нед × 40 ч) = ~289 руб/час

### Типовые коэффициенты для оценки длительности этапов

| Этап | Доля от общего срока (ориентир) |
|------|--------------------------------|
| Инициация | 3-5% |
| Анализ/проектирование | 7-10% |
| Инфраструктура | 8-12% |
| Основная разработка | 30-40% |
| ML/CV-модуль (если есть) | 15-20% |
| UI/Frontend | 10-15% |
| Интеграции | 5-8% |
| Тестирование | 8-12% |
| Пилот | 8-10% |
| Запуск и сдача | 3-5% |

**Буфер:** добавляй 10-15% к итоговому сроку и бюджету на непредвиденные риски.

## Шаг 5 — Сохранение и итог

Сохранить файл: `./Календарный_план_[slug]_v1.md`

Вывести итог:
```
ПЛАН ГОТОВ
---
Файл:          ./Календарный_план_[название]_v1.md
Срок:          [N] недель ([M] месяцев)
Бюджет:        [сумма] руб
Трудоёмкость:  [часов]
Этапов:        [N]
---
Следующие шаги:
  • Уточни даты старта → пересчитаю конкретные числа
  • Добавь реальные ставки → пересчитаю бюджет по этапам
  • Согласуй с командой → сохрани как v2
```

## РЕЖИМ 2: Доработка существующего плана

**Триггеры:** «пересчитай бюджет», «добавь этап», «измени сроки», «обнови часы», «переработай риски».

1. Прочитать файл (Read)
2. Применить изменение через Edit
3. Пересчитать итоги и помесячный бюджет
4. Если изменение > 20% — сохранить как v(N+1)
5. Подтвердить: «Обновлено. Новые итоги: [срок] / [бюджет]»
